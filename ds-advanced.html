从"会建模"到能交付的
中阶跃迁路径

这不是基础知识复习，也不是前沿论文导航。它回答一个更现实的问题：一个已经能做分析和建模的 DS，下一步如何把模型、实验和 LLM 应用做成可靠的生产系统。

ML Engineering LLM Applications Experimentation Production Readiness
使用方式：先做第 00 节自检，再按第 01 节选择一条主轨推进。第 02-05 节可作为工作中的参考手册：模型上线、RAG 评估、实验不显著、线上指标变差时，直接跳到对应模块查方法。
00

进阶前自检

如果这些项大多能独立完成，就可以开始读本页；否则先回基础页补齐关键短板

数据分析基础
能用 SQL 窗口函数写留存、漏斗、队列分析，并解释业务含义。
能做完整 EDA：缺失、异常、分布、分组差异，并提出下一步假设。
理解 p-value、置信区间、效应量的区别，不把"显著"当作"重要"。
能把业务问题拆成指标口径、观测窗口、用户分群和对照逻辑。
建模与实验
能训练 XGBoost/LightGBM，做交叉验证、调参和特征重要性解释。
能根据业务成本选择 precision/recall/F1/AUC，而不是只看最高分。
能设计基础 A/B 实验：样本量、随机单元、主指标和护栏指标。
知道数据泄漏、训练/测试污染、时间穿越特征会让离线结果虚高。
工程基础
能管理 Python 环境和依赖，知道 notebook 到脚本的基本组织方式。
能使用 Git 分支、commit、diff，知道如何复现一次实验代码。
知道模型持久化、配置文件、日志、异常处理的基本用途。
能读懂一个简单 API 服务如何接收请求、调用模型、返回结果。
大部分勾选：继续读本页。你的下一个瓶颈通常不是模型算法，而是数据管道、评估体系、上线稳定性和跨团队协作。
勾选少于一半：建议先回基础页补统计、SQL、ML 评估和实验设计，否则后面的工程化内容会变成背概念。
01

进阶路线图

不是所有方向都要同时学。优先选择一条主轨，按冲刺节奏做出可展示的产物

ML 工程化轨
8 周 · 优先级最高 · 从 notebook 到生产
1-2周重构一个旧模型项目：加 schema check、数据分布断言、特征生成脚本。
3-4周接入 MLflow：记录数据版本、参数、指标、模型 artifact 和关键图表。
5-6周用 FastAPI 封装模型服务，定义请求 schema、响应 schema、p95 latency 目标。
7-8周建立监控面板：数据漂移、模型指标、业务指标、服务指标分层展示。
LLM 应用轨
6 周 · 从 POC 到可靠 RAG/Agent
第1周实现一个小型 RAG：文档切块、embedding、向量检索、引用来源展示。
第2周加入 BM25 + reranker，对比纯向量检索和混合检索的召回差异。
3-4周建立评估集：20 个高频问题、10 个边界问题、5 个故意无答案问题。
5-6周加入 trace、成本统计、prompt 版本管理，尝试 LangGraph 工具调用流程。
分析深化轨
6 周 · 从"有结果"到"结果可信"
1-2周实现 CUPED：选实验前协变量，比较方差减少前后的置信区间宽度。
3-4周用 EconML 做一次 DML 估计，理解 nuisance model 和置信区间。
5-6周设计一个指标树：OEC、护栏指标、诊断指标、反作弊指标各自分层。
推荐顺序：如果你当前没有明确项目，先走 ML 工程化轨；如果你正在做 AI 应用，先走 LLM 应用轨；分析深化轨可以并行做，但不要替代工程能力训练。
02

ML 工程化实战

从 notebook 到生产系统：数据、训练、服务、监控四个环节缺一不可

Training-serving skew

离线训练和线上服务使用了不同的特征计算逻辑，就会出现 training-serving skew。最常见原因是 notebook 里临时处理了缺失值、线上服务用了不同时间窗口、特征更新频率不一致。

高风险线上指标骤降

是否需要 Feature Store

不是所有团队都需要 Feature Store。如果特征数量少、全是离线批处理，脚本+数据契约足够；如果同一特征被多个模型复用，且有在线低延迟需求，就应该考虑 Feast/Tecton。

判断标准复用度 × 实时性
上线前核对：特征定义是否版本化？训练和线上是否调用同一份转换代码？窗口边界是否包含未来信息？缺失值策略是否一致？日志里能否回放一次线上预测？
常见坑：离线用全量数据做 target encoding，线上只看历史；离线按自然日聚合，线上按滚动 24 小时聚合；实验组和对照组使用了不同特征刷新频率。

数据验证

训练前先跑 schema check、取值范围断言、类别枚举检查和分布漂移检测。数据验证不是 QA 的额外步骤，而是训练管道的入口门。

MLflow 追踪

重点记录：数据版本、特征版本、参数、指标、模型 artifact、重要图表和 commit hash。不要把每个中间变量都追踪进去，否则 experiment UI 会变成垃圾场。

模型 Registry

模型进入 registry 才算候选上线版本。每个版本至少应有 stage、训练数据��间窗、主指标、负责人、回滚说明。

最低可用训练管道：load data → validate data → build features → train → evaluate → register model。每一步都应该可单独重跑，并保留输入输出。
服务模式适用场景核心风险
批量预测每日推荐名单、营销人群、风险评分数据延迟、结果过期、回填失败
实时服务搜索排序、风控、个性化推荐p95/p99 latency、依赖超时、特征缺失
流式服务实时特征、异常检测、在线学习状态一致性、乱序事件、checkpoint 恢复
发布流程：先 shadow deployment 只记录不影响用户，再 canary 给 1-5% 流量，观察系统指标和业务护栏，最后逐步放量。上线模型一定要有快速回滚路径。
Latency SLO：不要只看平均耗时。排序/推荐/风控服务应至少看 p50、p95、p99；p99 的长尾通常来自外部依赖和冷启动。
层级监控内容告警方式
系统指标QPS、错误率、p95/p99 latency、CPU/GPU实时告警
数据指标缺失率、分布漂移、类别新值、特征 freshness小时/日报
模型指标AUC、precision、recall、calibration、分群表现日报+异常告警
业务指标转化率、留存、GMV、投诉率、护栏指标实验看板
漂移指标选择：PSI 适合业务看板解释；KS 适合数值分布差异；Wasserstein 能衡量分布移动距离。不要只靠一个漂移数值做自动回滚。
03

LLM 应用工程

LLM 应用不是把 API 接起来，而是检索、评估、可观测性和安全边界的系统工程

RAG 工程

切块策略决定召回上限。Fixed-size 简单稳定，sentence-based 更自然，semantic chunking 适合长文档但成本更高。生产 RAG 通常需要混合检索：向量检索负责语义相似，BM25 负责精确关键词，再用 reranker 重排。

  • 不要只用向量检索，数字、专有名词、代码名会丢召回。
  • 长上下文不是万能，"lost in the middle"会让中间证据被忽略。
  • 答案必须带引用来源，否则无法审计幻觉。

评估体系

RAGAS 的四个核心维度可以作为起点：Faithfulness 看答案是否忠于上下文，Answer Relevancy 看是否回答了问题，Context Precision 看召回内容是否噪声少，Context Recall 看证据是否足够。

  • LLM-as-Judge 要随机化答案位置，减少位置偏差。
  • 评估集必须包含无答案问题，测试模型是否会胡编。
  • 关键 query 应进入 CI，每次 prompt 或检索改动都跑回归。

Agent 生产化

Agent 的风险来自工具访问权。每个工具都应最小权限、参数可验证、操作幂等、设置超时。失败处理不要只靠 retry：能恢复就 retry，能降级就 fallback，涉及资金/权限/外部状态变更就 escalate-to-human。

  • 必须记录 trace：模型输入、工具调用、输出、耗时、成本。
  • 关键工具调用要有人审或规则审。
  • Prompt 版本必须和线上结果绑定。
LLM 应用上线 8 项检查
有固定评估集，覆盖高频、边界、无答案、对抗输入。
有 trace、成本、latency、成功率、人工接管率监控。
输出格式经过结构化验证，不合法时有 fallback。
外部工具最小权限，不允许模型直接执行不可逆操作。
Prompt、检索参数、模型版本都可回滚。
知道单次请求成本和每日预算上限。
04

实验与度量深化

中阶 DS 的分水岭：不是跑出显著，而是知道结果为什么可信或不可信

问题现象根本原因解法
SRM：实验组比例不对分流逻辑 bug、去重口径不一致、日志丢失先停读实验结果；用 chi-square 检测；复查曝光日志和随机单元
首日效果特别大新奇效应或学习效应延长观察窗口，拆分 day 1 / day 7 / day 14 效果
每天看结果直到显著p-hacking，提高假阳性率预先固定分析时间，或使用序贯检验
对照组也被影响网络效应或市场供需干扰Cluster randomization、switchback experiment、准实验方法
很多指标总有一个显著多重比较问题预注册主指标，次级指标只做诊断，必要时校正 p-value
主指标涨，护栏指标跌局部优化损害长期价值设 OEC + guardrail，推动产品决策而非只报 p-value
CUPED 公式：选择实验前指标 X，计算 θ = cov(Y, X) / var(X)，然后用 Y' = Y - θ(X - mean(X)) 作为调整后的结果。随机分组保证不会引入偏差，但协变量必须在实验开始前观测。
theta = cov(y, x_pre) / var(x_pre) y_cuped = y - theta * (x_pre - x_pre.mean()) effect = y_cuped[treatment].mean() - y_cuped[control].mean()
实验不可行时：样本不够 → 用 CUPED/延长周期/改更敏感指标；不能随机 → 考虑 DML/匹配/IV；单元不可分或有干扰 → 考虑合成控制或 switchback。
05

问题解法手册

工作中遇到问题时先定位类型，再选择最小可行修复，不要盲目换模型

问题诊断方向快速解法跳转
线上 AUC 明显低于离线training-serving skew、数据泄漏、时间切分错误回放线上请求，对齐特征生成逻辑#mlops
模型预测突然变差数据漂移、特征刷新失败、上游 schema 变化先查数据质量 dashboard 和特征 freshness#mlops
A/B 实验跑不显著方差大、样本小、指标太迟钝CUPED、延长周期、换更敏感代理指标#experimentation
RAG 答案不准确召回差、context 噪声、模型幻觉混合检索 + reranker + faithfulness eval#llmapp
LLM 输出格式不稳prompt 约束弱、没有验证层JSON mode / schema validation / fallback#llmapp
Feature 数量爆炸无治理、重复特征、无 owner特征重要性过滤、owner 标注、Feature Store#mlops
LLM 失效
幻觉

模型生成看似合理但没有证据支持的内容。

防御：强制引用来源，faithfulness eval，无证据时输出"不知道"。
LLM 失效
召回失败

正确答案在知识库里，但检索阶段没有取出来。

防御：混合检索、query rewrite、reranker、召回集人工抽检。
LLM 失效
Prompt 注入

外部文本诱导模型忽略系统指令或滥用工具。

防御：工具最小权限、隔离不可信内容、关键操作人工确认。
LLM 失效
成本失控

长上下文、多轮 retry、推理模型路由不当导致 token 成本飙升。

防御：缓存、模型路由、context 压缩、每用户预算上限。
