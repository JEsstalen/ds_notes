00.5

核心概念速查

在读专家页之前，先把这些高频词语理解到"能用自己的话解释"的程度

专家页假设你已经知道这些词是什么意思。这里不做深度推导，只帮你建立直觉——读完专家页的技术内容时，每遇到下面某个词，心里有个"对，我知道这是什么"的锚点就够了。
LLM 基础
Transformer / 注意力机制

现代所有大语言模型（GPT、Claude、Llama）的基础架构。核心思路：处理一段文字时，让模型对每个词都"看一遍"其他所有词，动态决定谁对谁更重要（这就是注意力权重）。相比之前的 RNN 逐步读入，Transformer 可以并行处理整段输入，训练效率大幅提升。
为什么要知道：专家页的 FlashAttention、MoE、长上下文等内容都建立在 Transformer 上，不理解注意力就无法理解它们优化的是什么。

专家页深度：发展脉络 2017 · FlashAttention 论文卡
LLM 基础
预训练 / 微调 / RLHF

预训练：在海量文本上学习语言规律（"下一个词是什么"）——代价极高，只有大厂做。
微调（Fine-tuning）：在预训练模型上继续训练，让它专注某个任务或风格——DS 可以做。
RLHF（人类反馈强化学习）：让人类标注"哪个回答更好"，用这个偏好信号调整模型行为——ChatGPT 能"听话"的原因。
为什么要知道：专家页的 DPO、GRPO、LoRA 都是微调/对齐方法的变种，知道三步骤才能理解它们各自省掉了什么。

专家页深度：发展脉络 2021-2022 · Tier 1 InstructGPT
LLM 基础
RAG（检索增强生成）

让 LLM 回答问题前，先从知识库里检索相关段落，把它塞进 prompt，再让模型作答。好处是答案有依据可验证、知识可更新、幻觉率低。
比喻：LLM 是开卷考试，RAG 是帮它找到要翻的那页书，而不是靠死记。
为什么要知道：专家页技术雷达里 RAG Pipeline 是 Adopt 级别，LLM 落地最主流的架构，本进阶页第 03 节会深入工程细节。

本页深度：LLM 应用工程 §03 · 专家页：技术雷达
LLM 基础
Prompt Engineering vs. Fine-tuning

Prompt Engineering：通过精心设计输入文字来引导模型输出，不改变模型权重，零成本但效果有上限。
Fine-tuning（微调）：用你自己的数据继续训练模型，改变权重，成本更高但能深度定制风格/知识。
什么时候选哪个：先试 Prompt；如果 Prompt 已经到上限（格式总不稳、专有术语总出错、需要大量 few-shot 例子），再考虑 LoRA/QLoRA 微调。

专家页深度：技术雷达 LoRA/QLoRA (Adopt) · 手工 Prompt 调优 (Hold)
数据工程
Lakehouse / 表格式（Iceberg）

传统数据湖（便宜、灵活但查询慢）和数据仓库（快速可查询但封闭昂贵）的折中。Lakehouse 在廉价对象存储上加了一层元数据，支持 ACID 事务（写入不会搞坏正在查询的人）、时间旅行（查昨天的数据）、schema 变更。
Apache Iceberg 是 2025 年赢得"表格式战争"的标准，Snowflake、Flink、Spark 都支持。
为什么要知道：专家页工具生态里 Apache Iceberg 是 Adopt，现代数据团队的基础设施选型。

专家页深度：工具生态 数据存储 · Lakehouse 论文
因果推断
相关 vs. 因果 / 混淆变量

冰淇淋销量和溺水死亡人数正相关——但冰淇淋不会导致溺水，两者的共同原因是夏天（混淆变量）。
DS 工作中，直接回归得到的系数通常是"相关关系"，不能直接用于决策。比如"发优惠券的用户购买率高"可能只是因为本来就倾向购买的人才会去领券。
为什么要知道：专家页的 DML、CUPED、DID、合成控制都是解决"怎么从相关性里提取因果效应"这一问题的工具。

本页深度：因果方法选择指南 §04.5 · 专家页：因果推断论文
因果推断
DID / 合成控制 / DML

DID（差分中差）：比较处置组和对照组在干预前后的变化差，抵消不随时间变的混淆。假设：如果没干预，两组趋势应该平行。
合成控制：找不到好对照组时，用加权平均构造一个"假如没有干预会怎样"的反事实。
DML（双重去偏机器学习）：高维特征场景下，用 ML 先把混淆因素的影响去掉，再做因果估计，比传统 OLS 更鲁棒。
为什么要知道：专家页 Tier 1/2 论文都围绕这三类方法展开。

本页深度：因果方法选择指南 · 专家页：因果论文精选
MLOps
Training-Serving Skew / 数据漂移

Training-serving skew：训练时用了一种特征计算逻辑，线上服务用了另一种——结果模型拿到的输入和训练时不一样，性能骤降。最常见原因是 notebook 里随手处理缺失值、线上忘了同样处理。
数据漂移（Data Drift）：随着时间推移，真实世界的数据分布变了（比如用户行为模式变了），但模型没更新。现象：离线指标好、线上越来越差。
为什么要知道：专家页 MLOps 论文和工具（Evidently AI）都在解决这两类问题。

本页深度：ML 工程化实战 §02 · 专家页：监控/可观测性工具
MLOps
MCP / Agent / LangGraph

AI Agent：不只是回答问题的 LLM，而是能调用工具（搜索、写文件、执行代码）自主完成多步任务的系统。
MCP（模型上下文协议）：Anthropic 发布的开放标准，定义 Agent 如何与外部工具/数据源交互——类比 USB 接口，不同工具用同一套接口。
LangGraph：基于状态机的 Agent 编排框架，适合有条件分支和人工介入的生产 Agent，2025 年的事实标准。
为什么要知道：专家页技术雷达 MCP 和 LangGraph 都是 Adopt，是 2026 年 Agent 工程的基础层。

本页深度：LLM 应用工程 §03 · 专家页：技术雷达 MCP/LangGraph

进阶路线图

不是所有方向都要同时学。优先选择一条主轨，按冲刺节奏做出可展示的产物

ML 工程化轨
8 周 · 优先级最高 · 从 notebook 到生产
1-2周重构一个旧模型项目：加 schema check、数据分布断言、特征生成脚本。
3-4周接入 MLflow：记录数据版本、参数、指标、模型 artifact 和关键图表。
5-6周用 FastAPI 封装模型服务，定义请求 schema、响应 schema、p95 latency 目标。

LLM 应用轨
6 周 · 从 POC 到可靠 RAG/Agent
第1周实现一个小型 RAG：文档切块、embedding、向量检索、引用来源展示。
第2周加入 BM25 + reranker，对比纯向量检索和混合检索的召回差异。
3-4周建立评估集：20 个高频问题、10 个边界问题、5 个故意无答案问题。

分析深化轨
6 周 · 从"有结果"到"结果可信"
1-2周实现 CUPED：选实验前协变量，比较方差减少前后的置信区间宽度。

推荐顺序：如果你当前没有明确项目，先走 ML 工程化轨；如果你正在做 AI 应用，先走 LLM 应用轨；分析深化轨可以并行做，但不要替代工程能力训练。
02

ML 工程化实战

从 notebook 到生产系统：数据、训练、服务、监控四个环节缺一不可

基础版失效点基础版教你用交叉验证评估模型——但那是在一份静态数据集上的评估。现实是：数据每天在变，特征计算逻辑在训练和服务之间可能不一致，模型"评估好"不代表"上线好"。本节解决的是离线指标和线上指标之间的那条鸿沟。
贯穿案例信贷风控模型上线：你在 notebook 里训练了一个 AUC 0.91 的 LightGBM 模型，使用了"用户近 30 天平均还款额"等聚合特征。本节讲的每个问题——training-serving skew、特征 freshness、模型 Registry、灰度发布——都对应这个模型从 notebook 走向真实信贷决策系统的一步。

Training-serving skew

离线训练和线上服务使用了不同的特征计算逻辑，就会出现 training-serving skew。最常见原因是 notebook 里临时处理了缺失值、线上服务用了不同时间窗口、特征更新频率不一致。

高风险线上指标骤降

是否需要 Feature Store

不是所有团队都需要 Feature Store。如果特征数量少、全是离线批处理，脚本+数据契约足够；如果同一特征被多个模型复用，且有在线低延迟需求，就应该考虑 Feast/Tecton。

判断标准复用度 × 实时性
上线前核对：特征定义是否版本化？训练和线上是否调用同一份转换代码？窗口边界是否包含未来信息？缺失值策略是否一致？日志里能否回放一次线上预测？
常见坑：离线用全量数据做 target encoding，线上只看历史；离线按自然日聚合，线上按滚动 24 小时聚合；实验组和对照组使用了不同特征刷新频率。

数据验证

训练前先跑 schema check、取值范围断言、类别枚举检查和分布漂移检测。数据验证不是 QA 的额外步骤，而是训练管道的入口门。

MLflow 追踪

重点记录：数据版本、特征版本、参数、指标、模型 artifact、重要图表和 commit hash。不要把每个中间变量都追踪进去，否则 experiment UI 会变成垃圾场。

模型 Registry

模型进入 registry 才算候选上线版本。每个版本至少应有 stage、训练数据��间窗、主指标、负责人、回滚说明。

最低可用训练管道：load data → validate data → build features → train → evaluate → register model。每一步都应该可单独重跑，并保留输入输出。
服务模式适用场景核心风险
批量预测每日推荐名单、营销人群、风险评分数据延迟、结果过期、回填失败
实时服务搜索排序、风控、个性化推荐p95/p99 latency、依赖超时、特征缺失
流式服务实时特征、异常检测、在线学习状态一致性、乱序事件、checkpoint 恢复
发布流程：先 shadow deployment 只记录不影响用户，再 canary 给 1-5% 流量，观察系统指标和业务护栏，最后逐步放量。上线模型一定要有快速回滚路径。
Latency SLO：不要只看平均耗时。排序/推荐/风控服务应至少看 p50、p95、p99；p99 的长尾通常来自外部依赖和冷启动。
层级监控内容告警方式
系统指标QPS、错误率、p95/p99 latency、CPU/GPU实时告警
数据指标缺失率、分布漂移、类别新值、特征 freshness小时/日报
模型指标AUC、precision、recall、calibration、分群表现日报+异常告警
业务指标转化率、留存、GMV、投诉率、护栏指标实验看板
漂移指标选择：PSI 适合业务看板解释；KS 适合数值分布差异；Wasserstein 能衡量分布移动距离。不要只靠一个漂移数值做自动回滚。
03

LLM 应用工程

LLM 应用不是把 API 接起来，而是检索、评估、可观测性和安全边界的系统工程

基础版失效点基础版教你 RAG 是什么、提示工程怎么写——在 demo 里效果不错。真实用户的问法更短、更口语、充满内部缩写和指代词，知识库里也有大量噪声文档。"demo 好用"和"生产可靠"之间，隔着评估体系、混合检索和可观测性。
贯穿案例电商客服 RAG 系统：产品经理想用 LLM 处理"我的订单什么时候到""怎么退款""优惠券能不能叠加"等 80% 的重复问题。本节讲的 RAG 工程、评估体系和 Agent 生产化，对应的就是这个系统从 demo 走向每天处理 10 万次对话的过程——以及它在真实用户面前为什么会失败。

RAG 工程

切块策略决定召回上限。Fixed-size 简单稳定，sentence-based 更自然，semantic chunking 适合长文档但成本更高。生产 RAG 通常需要混合检索：向量检索负责语义相似，BM25 负责精确关键词，再用 reranker 重排。

  • 不要只用向量检索，数字、专有名词、代码名会丢召回。
  • 长上下文不是万能，"lost in the middle"会让中间证据被忽略。
  • 答案必须带引用来源，否则无法审计幻觉。

评估体系

RAGAS 的四个核心维度可以作为起点：Faithfulness 看答案是否忠于上下文，Answer Relevancy 看是否回答了问题，Context Precision 看召回内容是否噪声少，Context Recall 看证据是否足够。

  • LLM-as-Judge 要随机化答案位置，减少位置偏差。
  • 评估集必须包含无答案问题，测试模型是否会胡编。
  • 关键 query 应进入 CI，每次 prompt 或检索改动都跑回归。

Agent 生产化

Agent 的风险来自工具访问权。每个工具都应最小权限、参数可验证、操作幂等、设置超时。失败处理不要只靠 retry：能恢复就 retry，能降级就 fallback，涉及资金/权限/外部状态变更就 escalate-to-human。

  • 必须记录 trace：模型输入、工具调用、输出、耗时、成本。
  • 关键工具调用要有人审或规则审。
  • Prompt 版本必须和线上结果绑定。
LLM 应用上线 8 项检查
有固定评估集，覆盖高频、边界、无答案、对抗输入。
有 trace、成本、latency、成功率、人工接管率监控。
输出格式经过结构化验证，不合法时有 fallback。
外部工具最小权限，不允许模型直接执行不可逆操作。
Prompt、检索参数、模型版本都可回滚。
知道单次请求成本和每日预算上限。
03.5

LLM 核心原理：从会调用 API 到看懂论文

不用推公式，但要理解 expert 页里那些方法到底在解决什么问题

学习目标：读完这一节，不要求你能训练大模型，但要能说清楚：为什么 Transformer 能工作、为什么 RAG 能减少幻觉、为什么 LoRA 能省显存、为什么推理模型更贵但更强。

1. Token 与上下文窗口

LLM 不直接读"字"或"词"，而是读 token。中文一个字可能是一个 token，英文一个词可能拆成多个 token。上下文窗口就是模型一次能看到的 token 数。长上下文不是越长越好：输入越长成本越高，模型也可能忽略中间证据（lost in the middle）。

  • 工程判断：能检索就不要盲目塞全文。
  • expert 连接：FlashAttention 优化的是长上下文注意力的计算/显存瓶颈。

2. Embedding 与向量检索

Embedding 把文本变成一串数字向量，语义相近的文本向量距离更近。RAG 的第一步通常是把知识库切块后做 embedding，用户问题也做 embedding，再找距离最近的片段。

  • 坑：向量相似不等于关键词匹配，产品名、数字、代码、时间容易漏。
  • 工程判断：生产 RAG 通常要混合检索：vector + BM25 + reranker。

3. LoRA / QLoRA 为什么省钱

全量微调要更新模型所有参数，显存和数据成本很高。LoRA 的想法是：大模型原来的权重不动，只在关键矩阵旁边加一小组低秩参数，让这小组参数学习你的任务。QLoRA 再把底座模型量化到 4-bit，进一步降低显存。

  • 适合：专有格式、稳定语气、领域术语、分类/抽取任务。
  • 不适合：事实知识频繁更新的问答；这类更适合 RAG。

4. DPO / GRPO 在优化什么

SFT 让模型学会"按示例回答"，但不一定知道哪个回答更好。DPO 用成对偏好数据（A 比 B 好）直接优化模型偏好，比 PPO/RLHF 简化很多。GRPO 是 DeepSeek-R1 路线里用于推理训练的强化学习方法，核心是让模型通过多次尝试比较相对好坏，学习长链推理。

  • DPO：更像"偏好微调"，适合对齐风格/安全/有用性。
  • GRPO：更像"训练模型思考过程"，适合数学、代码、可验证推理。

5. Test-Time Compute 为什么贵

普通模型一次生成答案；推理模型会在回答前生成更长的思考链、尝试更多路径、做自我检查。能力提升来自"推理时花更多算力"，不是模型凭空变聪明。代价是 token 数、延迟和成本都大幅增加。

  • 工程判断：不要所有问题都用推理模型；先做路由，简单问题走快模型，复杂问题才走 reasoning model。
  • expert 连接：专家页的 Test-Time Compute 和成本曲线都围绕这个权衡。

6. LLM-as-Judge 为什么有偏差

用强模型评估弱模型输出很实用，但它不是客观真理。常见偏差：更喜欢长答案（冗长偏差）、更喜欢第一个展示的答案（位置偏差）、更喜欢自己家模型的风格（自我偏好）。

  • 工程判断：评估时随机化答案顺序；关键样本人工复核；不要只看 judge 分数。
  • expert 连接：专家页的 LLM-as-Judge 论文卡专门讲这些偏差。
你要解决的问题优先方案不要先做什么为什么
回答企业文档/知识库问题RAG直接微调知识会更新，RAG 可溯源、可替换文档；微调记知识成本高且容易过期。

04

实验与度量深化

中阶 DS 的分水岭：不是跑出显著，而是知道结果为什么可信或不可信

基础版失效点基础版教你设计 A/B 实验、计算 p-value——你能跑出一个"显著"的结果。但显著不等于真实：SRM 让你的分流就已经坏了、首日效应骗了你、每天看结果让假阳性飙升、护栏指标没设导致发布后用户投诉。这节教你区分"看起来有效"和"真的有效"。
贯穿案例内容推荐 A/B 实验：你给 Feed 流推荐算法做了一次实验，点击率提升 8%，p=0.03，你准备发布。但——实验组和对照组比例是 52:48 不是 50:50（SRM）；你看了 3 天就停实验；次日留存没有设护栏；社区网络效应让对照组也被污染了。本节每个陷阱都对应这个决策链上的一个断点。
问题现象根本原因解法
SRM：实验组比例不对分流逻辑 bug、去重口径不一致、日志丢失先停读实验结果；用 chi-square 检测；复查曝光日志和随机单元
首日效果特别大新奇效应或学习效应延长观察窗口，拆分 day 1 / day 7 / day 14 效果
每天看结果直到显著p-hacking，提高假阳性率预先固定分析时间，或使用序贯检验
对照组也被影响网络效应或市场供需干扰Cluster randomization、switchback experiment、准实验方法
很多指标总有一个显著多重比较问题预注册主指标，次级指标只做诊断，必要时校正 p-value
主指标涨，护栏指标跌局部优化损害长期价值设 OEC + guardrail，推动产品决策而非只报 p-value
CUPED 公式：选择实验前指标 X，计算 θ = cov(Y, X) / var(X)，然后用 Y' = Y - θ(X - mean(X)) 作为调整后的结果。随机分组保证不会引入偏差，但协变量必须在实验开始前观测。
theta = cov(y, x_pre) / var(x_pre) y_cuped = y - theta * (x_pre - x_pre.mean()) effect = y_cuped[treatment].mean() - y_cuped[control].mean()
样本量估算参考
效应量（MDE）基础转化率所需单侧样本量（α=0.05, power=0.8）实际意义
5% 相对提升10%~30,000 / 组大厂核心漏斗，效应小，需要大量样本
10% 相对提升10%~7,700 / 组中等效应，大多数产品实验的合理期望
20% 相对提升10%~2,000 / 组重大改版，效应大，样本量要求低
5% 相对提升2%~155,000 / 组低转化漏斗底部，极难做实验
常见误区：先跑实验，样本量不够就延长时间——但延长会遇到季节效应和 p-hacking 问题。正确顺序：先算样本量 → 确认实验周期 → 开始实验。Python 工具：statsmodels.stats.power.tt_ind_solve_power
实验不可行时：样本不够 → 用 CUPED/延长周期/改更敏感指标；不能随机 → 考虑 DML/匹配/IV；单元不可分或有干扰 → 考虑合成控制或 switchback。
04.5

因果方法选择指南

专家页里出现的 DML、DID、Synthetic DID、IV 在什么场景下该选哪个

基础版失效点基础版教你建回归模型预测结果——但预测不是因果。老板问"我们的会员计划真的提升了购买频次吗"，用回归系数回答是错的，因为愿意买会员的人本来就买得多。从"相关性分析"到"因果估计"，是一个思维框架的彻底升级。
贯穿案例会员激励计划评估：公司给一批用户发了优惠会员，三个月后这批人的 GMV 涨了 25%。这是会员计划的效果，还是因为本来就爱买的人更容易接受会员？本节的 DID、DML、IV 方法，每一个都在尝试用不同假设回答这同一个问题。
选方法的核心问题：你能不能做随机实验？如果不能，为什么不能？"为什么不能"决定了哪种准实验方法适合。

DID 的致命陷阱

标准 2×2 DID（只有处置前/后 × 处置组/对照组四个格子）在分阶段上线的产品中会严重偏估。原因：不同时间上线的用户批次本来就不同，OLS 把这个差异当作处置效应。

高风险分阶段上线场景

修复：用 Sun-Abraham 或 Callaway-Sant'Anna 方法，专家页 Tier 2 论文有详细说明。

为什么不先用 PSM（倾向得分匹配）

PSM 匹配后的"干净样本"看上去平衡，但在高维特征场景下会引入选择偏差——被匹配掉的样本可能恰好是有异质处置效应的群体。DML 通过正交化保证高维场景下的无偏性，是比 PSM 更统计严格的选项。

大厂 MLE / Senior DS 面试必有 1 轮：从零设计一个完整的 ML 系统——考察的是工程判断力，不是算法细节

面试定位：ML 系统设计轮没有标准答案，但有标准框架。面试官想听到的是：你如何从模糊需求出发，拆解问题、定义指标、做出有依据的设计取舍——而不是背一个架构图。

标签定义：正样本是什么

推荐系统的训练质量取决于标签定义。选项比较：

  • 点击：噪声大，标题党行为正向激励
  • 购买：稀疏，冷启动难
  • 观看时长 / 划走速度：内容推荐更合适
  • 加购 + 收藏 + 分享：多目标，需要加权
常见错误只用点击当正样本

Two-Tower 模型原理

用户塔（User Tower）和物品塔（Item Tower）各自独立编码，输出向量，用内积衡量匹配度。

  • 优点：物品向量可离线预计算，线上只需查表 + 内积，延迟极低
  • 缺点：用户-物品交叉特征在塔内无法捕获（精排才做交叉）
  • 负采样：随机负采样 + 热门 item 降权（避免模型偏向流行内容）

冷启动解法

三种冷启动场景的标准处理：

  • 新用户：基于注册信息 + 新手问卷 → 人口属性画像召回
  • 新 item：用 item 内容特征（图、文、类目）做 Content-based 召回；设置强制曝光保底
  • 新场景：迁移学习（从相近场景的模型权重初始化）

问题定义主动澄清需求，说出多个可能的方向并解释为什么选这个直接开始画架构图，没有澄清
指标设计离线指标、在线指标、护栏指标分层说清，能说出每个指标为什么选只说 AUC，或只说 CTR
数据意识主动提标签噪声、position bias、数据分布偏移、冷启动把数据当已知，只讨论模型
工程判断能说出"在这个延迟要求下，XX 方案不可行，因为……"设计脱离工程约束（99.9th latency 要求下跑 BERT）
取舍能力主动说出这个设计的限制，以及在什么条件下会换方案认为自己的方案没有缺点
沟通清晰度先说结论，再说推理；用类比解释复杂选择边说边想，没有结构
一个能显著加分的习惯：每做一个设计决策，说出"我在这里做了一个取舍：A 换来了 B，代价是 C——在当前场景下 B 比 C 更重要，所以选 A"。这个句式让面试官看到的不是答案，而是你的思考过程。
05.5

综合实战项目：从 notebook 到生产级系统

把本页所有模块——ML 工程化、LLM 应用、实验设计、因果推断——串成一个可展示的完整系统

项目定位：这不是比谁的模型 AUC 最高。面试官想看到的是：你能不能把一个模型做成可信赖的生产系统——有监控、有评估、有回滚机制、知道什么时候该重训练。做完这个项目，你就有了一个完整的 MLE 面试案例。
衔接基础版基础版的综合实战项目止步于"模型评估和 SHAP 解释"。本项目从那里出发：同样的信贷违约预测场景，下一步是把那个 notebook 做成一个工程化、可监控、可验证因果效应的系统。
贯穿案例信贷违约系统上线：基础版给了你一个 AUC 0.91 的 LightGBM 模型和三张 SHAP 图。现在业务要求把它接入审批流程，每天处理 5000 笔申请。本项目每个阶段都在回答同一个问题：这个模型在线上真的靠谱吗？
Phase 1 · ML 工程化

消除 training-serving skew

把基础版 notebook 里的特征计算逻辑抽成独立函数，确保训练和推理调用完全相同的代码路径。接入 MLflow 记录数据版本、特征版本和模型 artifact。

好结果：能给出一份"上线前核对清单"，并指出基础版 notebook 里至少 2 处潜在 skew 风险（例如 target encoding 在训练时用了全量数据）。MLflow experiment 截图可在面试中演示。
跳转ML 工程化实战 §02
Phase 2 · 监控体系

建立四层监控面板

对信贷模型，数据层监控"用户近 30 天还款额"等关键特征的分布漂移（PSI），模型层监控每日 AUC 和校准误差，业务层监控实际坏账率 vs. 模型预测坏账率。

好结果：一张截图或表格，展示四层指标各选了什么、为什么选这个、阈值怎么定。能说清楚：PSI > 0.2 触发的是"调查"还是"自动停用"，以及为什么。
跳转监控系统 Tab
Phase 3 · 实验验证

设计上线 A/B 实验

新模型不能直接全量替换旧规则。设计一个实验：主指标是 60 天实际坏账率，护栏指标是审批通过率（不允许下降超过 5%）。预计算样本量和实验周期。

好结果：一份实验设计文档（1页），说明：随机单元（申请 ID 还是用户 ID？）、主指标和护栏指标、最小可检测效果是多少、为什么是 60 天观察窗口而不是 7 天。
跳转实验与度量深化 §04
Phase 4 · 因果验证

证明模型带来了真实效果

实验跑完后，除了 p-value，还需要回答：新模型的效果是均匀的还是集中在某类用户？用 EconML CausalForest 估计异质处置效应（HTE），找出模型效果最大的细分群体。

好结果：一张 CATE 分布图 + 1 段业务解读——"新模型对月收入 5000-15000 的中等信用用户效果最显著（CATE = -8.3% 违约率下降），对高信用用户效果接近零，这意味着可以针对性地只对中等信用群体启用新模型"。
跳转因果方法选择指南 §04.5
最终交付物：一个 GitHub 仓库 + 一份系统设计文档（2-3 页），涵盖：数据管道设计、特征版本策略、监控指标选择理由、实验设计、HTE 分析结论。面试时能说出："我做的不只是一个模型，而是一个从数据更新到线上监控都可追溯的信贷评分系统"。
进阶版常见失误：监控只看 AUC 不看业务指标；实验上线没有灰度发布计划；因果分析把 ATE 直接当个体决策依据；MLflow 记录了太多无用变量导致实验对比困难。
